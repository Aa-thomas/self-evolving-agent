Project 1A · Primitive 7 · Trace Logger

A trace is replayable evidence

A terminal result tells you where an agent run ended. A causal trace preserves the authoritative transitions needed to explain how it got there.

A symptom is not a root cause

Open the real partial-run.json:

{
  "final_answer": null,
  "exit_reason": "max_steps"
}

Two facts are supportable: no answer was submitted, and the harness exhausted its budget. The artifact does not establish whether the model repeated valid work, emitted malformed JSON, selected an unavailable handler, encountered a tool exception, or followed a legitimate trajectory that needed more steps.

Central thesis: when recorded evidence cannot distinguish plausible first bad transitions, “evidence insufficient” is the correct diagnosis—not an invitation to blame the model by default.

Separate observations from explanations

Available observationQuestion it answersQuestion it cannot answer
final_answer: nullWas a final answer returned?What proposal or rejection prevented one?
exit_reason: max_stepsWhich stop condition ended the run?Whether those steps were repeated, rejected, failed, or useful.
No ordered stepsNothing about intermediate causality.Which component owns the first correction.

The smallest safe intervention is therefore to add evidence at the boundaries that already know the missing facts. Do not change parser, validator, dispatch, tool behavior, or budget policy until a future trace implicates one of them.

Commit the first missing evidence

A run has exit_reason="max_steps" but no assistant outputs. What exact diagnosis is impossible?

Without assistant output you cannot distinguish repetition, malformed JSON, or a changed request. Timing and storage may matter later, but neither reveals the first bad transition.

Record each fact where it becomes authoritative

The loop is a sequence of trust and control boundaries. Each one is the first place that can state a different fact without inference.

BoundaryAuthoritative factStable trace evidence
Model completionThe exact text the model proposed.assistant_output
Parser and validatorAcceptance or the first failed request stage.RequestAccepted or RequestRejected
Loop classifierWhether an accepted request means submit or execute.SubmitSelected or ExecuteToolSelected
Runtime dispatchWhether external execution succeeded or failed.ToolSucceeded or ToolFailed
Loop controlWhy the run stopped.submitted or max_steps

A final AgentResult cannot reconstruct discarded assistant text or intermediate request outcomes. Instrumenting only after the run would create a plausible story, not a causal record.

Runtime types and persisted evidence have different jobs

The live request pipeline uses Result[ValidatedToolRequest, typed errors] because those types make safe execution and exhaustive control flow easier. A persisted trace has a longer-lived responsibility: remain interpretable when runtime libraries and class names change.

runtime Result

RequestAccepted | RequestRejected
    → TraceStep

Inspect the supplied types and adapter in trace_logger.py. A rejection records the first failed RequestStage. Acceptance implies all earlier request gates succeeded. Explicit kind discriminators prevent a loose dictionary from representing contradictory states.

Tradeoff: the adapter adds a mapping layer. In return, replacing Result, renaming a Pydantic model, or reorganizing runtime actions does not silently redefine old trace files.

Follow a complete read-and-submit trace

  1. The initial message asks, “Read notes.txt.” The logger copies the exact state supplied to the first model call.
  2. The model emits a read_file request. The request is accepted, execution is selected, and the handler returns "lesson notes".
  3. Step 1 records assistant output → RequestAccepted → ExecuteToolSelected → ToolSucceeded. It has no exit reason.
  4. The next model call sees the tool observation and emits submit(answer="done").
  5. Step 2 records assistant output → RequestAccepted → SubmitSelected → no tool outcome → submitted.

The absence of a tool outcome on submit is correct evidence. Submission is internal control flow; fabricating a successful external invocation would make the trace less truthful.

Collect mutably; finalize immutably

Events arrive incrementally, so TraceLogger uses mutable working lists during the run. At finish, it freezes initial messages and steps into tuples inside an immutable AgentTrace, then writes a JSON object with schema_version: 1.

This separates two lifecycles: collection while facts arrive, and evidence after the run is complete. Immutability prevents casual mutation of the finalized in-memory snapshot. It does not by itself prove truthfulness, durable storage, atomic writes, redaction, or migration safety.

Distinguish protocol rejection from runtime absence

An unknown protocol tool fails during request validation:

RequestRejected(
  stage="tool_lookup",
  code="UNKNOWN_TOOL"
)

A protocol-known tool with no runtime handler follows a longer path:

RequestAccepted(tool="read_file")
→ ExecuteToolSelected(tool="read_file")
→ ToolFailed(tool="read_file", code="RUNTIME_TOOL_UNAVAILABLE")

Runtime dispatch emits RUNTIME_TOOL_UNAVAILABLE. The distinct persisted code and ordered variants preserve the difference from a protocol-unknown tool.

Implement the smallest evidence slice

Add the narrow trace hook to run_agent, then implement TraceLogger.start, record_step, finish, and write_json. Construct variants where the facts become authoritative. Do not reconstruct the story from the final result or persist raw runtime objects.

Run proof, then state its limit

When the test turns green, inspect the JSON you wrote. Explain one trajectory as assistant output → request outcome → action → tool outcome or exit. Identify the first bad transition and the boundary that owns it.

The proof establishes that one covered run crosses the runtime-to-trace boundary, preserves typed causal outcomes, freezes its completed snapshot, and writes versioned JSON. It does not prove redaction, retention, backward-compatible migrations, truthful external systems, model quality, or that every future branch records an event.

Transfer: add the smallest regression for a request accepted by the protocol registry but rejected because its runtime handler is absent. Assert RequestAccepted → ExecuteToolSelected → ToolFailed(code="RUNTIME_TOOL_UNAVAILABLE"), and assert that it is not a RequestRejected(stage="tool_lookup").
