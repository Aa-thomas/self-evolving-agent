Primitive 2 · Message State

Memory is the context you pass back in

A model call has no secret memory of the call before it. The harness creates continuity by passing an ordered record of what happened.

Why does an agent repeat work?

Imagine the model requests read_file. The tool succeeds and returns the file contents. On the next turn, the model asks for read_file again.

It is tempting to blame the model: perhaps its temperature is too high or it ignored the result. But there is a simpler cause to check first. Did the next model call actually receive the tool result?

First principle: an agent's working state is the ordered message list supplied to the next model call. If an event is absent from that list, the model cannot use it.

Primitive 1 records one provider call. Primitive 2 connects calls by turning each relevant event into a model-visible message.

A message list is a causal ledger

Each message says who produced an event and what the next model call is allowed to know about it. Order matters because later events only make sense in relation to earlier ones.

RoleMeaning in this harnessExample from the real proof
userThe task or instruction that begins the trajectory.Read notes.txt.
assistantThe model output, including a proposed tool request.{"tool":"read_file",...}
toolThe harness's observation of what the tool returned.{"ok":true,"value":"file contents"}

The tool message is not the tool itself. It is evidence about the tool outcome, encoded as text so it can cross the model-call boundary.

What does “decode the tool observation” mean?

Inside the harness, a tool result may be a structured Python value such as {"ok": True, "value": "file contents"}. Message content must be a string, so append_tool_observation serializes that value to JSON and appends it with role="tool".

structured harness result
        ↓ json.dumps(...)
{"role": "tool", "content": "{\"ok\": true, ...}"}
        ↓ json.loads(message["content"])
structured observation for a test or diagnostic

decode_tool_observation performs the final arrow. It does not execute the tool and the model does not call it. It lets tests verify that the message string still contains the structured result the harness intended to preserve.

Contrast the missing transition

Suppose the tool result exists only in a local Python variable. If the harness calls the model again while the message list still ends with the assistant request, the second input is [user, assistant]. From the model's point of view, no tool result ever happened.

Assume hidden memory

Ask again and hope the model remembers an event that was never supplied. Repetition remains ambiguous.

Append explicit evidence

Record the tool observation before the next call. The input now explains what happened and preserves it for proof.

The harness chooses explicit evidence. This does not guarantee the model will act well, but it makes missing context distinguishable from poor model behavior.

Synthesis: state is a deliberate handoff

Nothing inside the model call automatically joins one turn to the next. Continuity exists only because the harness preserves an ordered event record and supplies it again. That makes message state both an input to behavior and an evidence artifact: the same list that gives the model context lets an engineer inspect what context was actually available.

The practical diagnostic rule is therefore simple. Before attributing repeated work to model judgment, inspect the next call’s exact message list. A missing or misordered observation is a harness failure; a complete and accurate list moves the investigation to a later boundary.
