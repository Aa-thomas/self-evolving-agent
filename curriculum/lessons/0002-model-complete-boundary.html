Make the model boundary an object with one method

Your next win is to understand what model.complete(messages) means before you write it: an object receives the current message list and returns raw assistant text.

The tiny contract

In the agent loop, model does not need to be a framework object. It only needs one behavior:

assistant_output = model.complete(messages)

Python's class tutorial explains that classes bundle data with functions, and that functions defined on a class become methods on instances. That is the only class idea you need here: put API configuration or fake outputs on self, then expose a complete method.

Boundary rule: complete talks to the model. run_agent decides what the returned text means.

Keep this reference open while you design: Model Completion Boundary.

What Python does for you

When you write a method like def complete(self, messages):, the first parameter is the instance. When you call model.complete(messages), Python supplies model as self. The Python docs call this the usual method call behavior.

class ToyResponder:
    def __init__(self, reply):
        self.reply = reply

    def complete(self, messages):
        return self.reply

model = ToyResponder("hello")
text = model.complete([{"role": "user", "content": "Say hi"}])

This toy class is not your final class. It shows the shape: __init__ stores state, complete uses that state, and callers use dot syntax.

Find the same shape in your tests

Your test file already contains the concept in miniature:

class FakeModel:
    def __init__(self, outputs):
        self.outputs = outputs
        self.calls = []

    def complete(self, messages):
        self.calls.append(deepcopy(messages))
        return self.outputs.pop(0)

That fake is useful because it proves run_agent should depend on behavior, not on a specific concrete class name. A real API-backed model and a fake test model can both satisfy the same .complete(messages) boundary.

Design before coding

Before editing agent_loop.py, answer these four questions on paper or in comments:

Question Why it matters
What should the object remember on self? Fake models remember outputs and calls. Real models remember client and model name.
What argument does complete accept? The loop passes the full message state, not just the newest user text.
What type does complete return? The parser expects raw assistant text, so the loop should get a string.
What should remain outside the model class? Parsing, tool validation, tool execution, stop conditions, and observations belong to the harness.

Practice: retrieve the boundary

Without looking above, write the method signature your loop expects and the one thing it should return.

The loop expects a method shaped like complete(self, messages). Calling it as model.complete(messages) should return raw assistant text as a string.

Now name three responsibilities that must stay in run_agent, not inside complete.

run_agent parses the returned text, validates or executes tool requests, and stops when the request is submit. The model class only produces text.
