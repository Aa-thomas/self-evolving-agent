Turn untrusted model text into one narrow request

The model proposes text. The harness decides whether that text earns the right to become structured data—and stops before claiming the request is safe to execute.

Why is JSON-looking text still only text?

Primitive 2 can preserve an assistant message such as {"tool":"read_file","args":{"path":"notes.txt"}}. To a person, it looks like a tool call. To the harness, it is still an untrusted string.

If later code treats any tool-looking string as an executable request, the model has silently gained permission to invent syntax, attach ignored prose, or smuggle a malformed value past an unclear boundary.

First principle: text becomes structured data only when an explicit parser establishes a small, testable contract. Looking structured is not the same as being parsed.

This primitive creates that first protocol boundary. It answers: “Is this exactly one JSON document with the outer shape of a ToolRequest?”

Build the mental model one claim at a time

The parser is easier to reason about when you refuse to collapse its two gates into one vague idea of “valid.”

StageQuestion it may answerClaim it still cannot make
Raw assistant textWhat exact characters did the model return?Whether those characters encode any JSON value.
parse_jsonIs this one complete JSON document?Whether the decoded value is a tool request.
validate_tool_requestIs the value an object with strict tool and args fields and no extras?Whether the named tool exists or its arguments match that tool.
Primitive 4Is the tool known and are its arguments valid for its schema?Whether a runtime handler is installed and succeeds.

Invariant: Primitive 3 returns either one ToolRequest, INVALID_JSON, or INVALID_TOOL_REQUEST_SHAPE. It never guesses a repair and never approves execution.

Walk through a successful request

Follow {"tool":"read_file","args":{"path":"notes.txt"}} through the real contract:

  1. json.loads consumes the entire string and returns a Python dictionary.
  2. ToolRequest.model_validate sees a strict string tool, a dictionary args, and no unexpected top-level fields.
  3. The parser returns Ok(ToolRequest(...)).

Notice the modest conclusion. The parser has proven the request's transport shape. It has not looked in a registry, checked that path is a string, opened a file, or granted permission.

Contrast the two failure families

Decoding fails

read_file(path="notes.txt"), two adjacent objects, malformed braces, and JSON followed by prose are not one complete JSON document. The first failed claim is decoding, so the result is INVALID_JSON.

Outer shape fails

"hello" and a top-level array containing one tool-like object are valid JSON values. {"tool":"read_file"} is a valid JSON object. None establishes the required ToolRequest shape, so the result is INVALID_TOOL_REQUEST_SHAPE.

The error code records the first claim that failed. It is not a general verdict that the whole agent run is bad.

Choose strictness deliberately

A permissive parser could accept Python-like calls, extract the first object from surrounding prose, or fill in a missing args field. That seems convenient because more model outputs continue through the loop.

The cost is hidden ambiguity: different spellings may mean different things, ignored suffixes disappear from evidence, and later components can no longer tell what the parser actually guaranteed.

Permissive guessing

Fewer immediate rejections, but a large undocumented language and surprising execution paths.

One strict protocol

One complete JSON document, one explicit outer model, and failures the loop can record and return as feedback.

This harness chooses the strict protocol. Recovery can happen visibly on a later turn; silent interpretation does not happen inside the parser.

Build from the scaffold

Copy the real 03_parse_tool_request_skeleton.py into your working 03_parse_tool_request.py. Implement only the three withheld functions.

  1. In parse_json, decode exactly one complete document and normalize only JSONDecodeError into InvalidJsonError.
  2. In validate_tool_request, use the supplied strict model and normalize its ValidationError into InvalidToolRequestShapeError.
  3. In parse_tool_request, stop on a decoding Err; otherwise pass the decoded value to the shape gate.

Do not add tool names, per-tool argument models, runtime handlers, retries, or prose extraction. Those would make this primitive claim more than it can prove.

Run proof, then inspect what you built

Run uv run pytest curriculum/phase-1/03_parse_tool_request_tests.py. Pay particular attention to test_trailing_prose_is_not_json, test_unknown_tool_name_still_produces_tool_request, and test_tool_specific_argument_type_is_left_for_validation.

The proof establishes

For the ten covered raw outputs, the implementation distinguishes decoding failure, outer-shape failure, and a structured request without swallowing suffixes or consulting later boundaries.

The proof does not establish

That a tool name is registered, its arguments are valid for that tool, a runtime handler exists, or execution is safe.
