Build a runner for composed behavior

You will implement the evaluator. The starter gives you a real multi-step workflow fixture, a report contract, and a red proof that becomes the evidence for your runner.

Inspect the workflow unit tests miss

Open eval-cases.json. The copy-file case requires three model outputs: read, observe, write, observe, then submit. The focused loop test test_tool_result_is_appended_before_next_model_call protects one transition; this fixture defines a user-visible workflow across several transitions.

Invariant: an eval result retains each case’s expected behavior, actual behavior, pass/fail decision, failure reason, and causal trace path. A count alone cannot diagnose a regression.

Commit the evaluation boundary

A refactor still passes the submit stop-condition unit test but writes the copied file with the wrong contents. What should catch it first?

The composed eval catches it: the unit test established that one observation is appended, not that the full read → write → submit workflow produced the desired file.

Implement the runner, not the answer key

Read eval_runner.py. Implement evaluate_case and run_eval_suite against the provided dataclasses. A case executes its scenario, compares actual and expected fields, records a focused failure reason, and preserves its trace path. Run uv run pytest curriculum/phase-1/08_eval_runner/tests_eval_runner.py; test_eval_runner_reports_each_case_and_causal_trace starts red until you construct the report.
