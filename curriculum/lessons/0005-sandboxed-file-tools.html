Turn a valid path into an authorized capability

Primitive 4 can prove that path is a string. Primitive 5 must decide whether the resource named by that string belongs to this run—and preserve exactly why an operation succeeds or stops.

A valid argument can still request forbidden authority

Consider ValidatedToolRequest(tool="read_file", args=ReadFileArgs(path="../secret.txt")). Its tool exists. Its required argument exists. The path is a string. Every claim made by protocol validation can be true while executing the request would still read outside the agent's working directory.

First principle: validating the shape of a resource name is not authorizing the resource it names. Authorization must happen where the concrete sandbox root and filesystem path are both known.

The sandbox is a capability boundary: it defines the portion of the filesystem this run is allowed to observe or change. A file tool must establish that boundary before existence checks, reads, writes, or listings can reveal anything.

Use a decision ladder, not one vague “safe path” check

A useful tool result names the first gate that could no longer make its claim.

GateQuestionRepresentative result
1. Input policyIs the path non-empty, relative, and free of parent-traversal syntax?EMPTY_PATH or FORBIDDEN_PATH
2. Canonical containmentAfter resolving aliases such as symlinks, is the target still beneath the resolved sandbox root?FORBIDDEN_PATH
3. Operation preconditionsDoes the target exist, have the right kind, and stay within the byte budget?FILE_NOT_FOUND, NOT_A_FILE, or CONTENT_TOO_LARGE
4. Filesystem effectDid the authorized read, write, or listing complete?Ok or a normalized operation error

Invariant: no file operation runs against a path outside the resolved sandbox root, while contained operations retain precise success and failure observations.

Walk through an allowed read

The real nested-file case creates sandbox/notes/today.txt and calls read_file("notes/today.txt", sandbox_root).

  1. The input is non-empty, relative, and contains no .. component.
  2. The root and requested path are resolved. The result remains relative to the resolved root.
  3. The target exists, is a file, and its encoded size is within MAX_FILE_BYTES.
  4. The tool reads UTF-8 text and returns Ok(value="nested evidence").

Containment is necessary, but it is not the entire tool contract. A contained missing path is not forbidden; it is FILE_NOT_FOUND. A contained directory passed to read_file is NOT_A_FILE.

Contrast two escape attempts

Visible traversal

../secret.txt contains an explicit parent component. The input-policy gate rejects it before the tool asks whether the outside file exists.

Alias traversal

linked.txt may look local while a symlink points outside. Resolving the target before the containment comparison exposes the escape and returns FORBIDDEN_PATH.

This is why checking only the characters in a path is insufficient. The file tool authorizes the resolved target, not the spelling alone.

Choose strict path policy deliberately

There are two plausible policies for notes/../safe.txt when the final target remains inside the sandbox:

Canonical containment only

Normalize the path and allow it if the final target is inside. More spellings work, but audit evidence contains aliases and traversal syntax.

Strict syntax plus containment

Reject every explicit .., then also resolve and check containment. The accepted language is smaller and every parent traversal is visible as a rejection.

Both still require canonical containment to catch symlinks. The repository makes one additional policy choice; commit to your prediction before checking its regression.

The proof establishes

Across twenty temporary-filesystem cases, the implementation preserves containment, distinguishes operation preconditions, bounds content, prevents covered outside writes, and returns stable Ok/Err observations.

The proof does not establish

It does not eliminate filesystem races, define every platform's symlink policy, prove arbitrary encodings are readable, or authorize future network and shell tools.
