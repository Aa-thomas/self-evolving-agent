Project 1A Primitive 5

Tools are permissioned code paths

read_file, write_file, and list_files are useful only because the harness restricts them to a sandbox root.

Why This Matters In Production

A file tool is not just a helper function. It is a permission boundary. If the model can trick the tool into reading ../secrets.env or writing outside the task directory, the agent has crossed from automation into a security incident.

Invariant: every filesystem path must resolve inside sandbox_root before the tool reads, writes, or lists anything.

Predict Before Reading

The request passed schema validation: {"tool": "read_file", "args": {"path": "../secret.txt"}}. Should the file tool still reject it?

Prediction: yes. Schema validation proves the argument has a path; it does not prove the path is safe for this sandbox.

Concrete Walkthrough

candidate = (sandbox_root / user_path).resolve()

if candidate is outside sandbox_root:
    return Err("Path escapes sandbox", "PATH_OUTSIDE_SANDBOX")

return read_text(candidate)

The implementation and proof live in 05_sandboxed_file_tools.py and tests_05_sandboxed_file_tools.py. Tool handlers call path validation immediately before filesystem access.

Failure Case

A customer asks the agent to summarize local notes. The model emits read_file("../customer_tokens.txt"). The request is syntactically valid and the tool name is known. The security boundary is enforced only if the file tool resolves the path and rejects anything outside the sandbox.

Evidence And Observability

Convincing evidence tests both allowed and forbidden paths. A good proof creates a file inside the sandbox, creates a tempting file outside it, then asserts the outside file is never read or written. The result should be a structured error, not a Python traceback.

Run the focused proof with uv run pytest curriculum/phase-1/05_sandboxed_file_tools/tests_05_sandboxed_file_tools.py.

Engineering Tradeoff

Flexible file paths make demos feel smoother, but strict sandboxing makes the agent safe to run repeatedly. Project 1A chooses explicit permissions over convenience: the model may request paths, but the tool owns the final filesystem decision.
