Project 1A Primitive 6

The agent loop owns control flow

Primitive 6 composes model calls, message state, parsing, validation, file tools, observations, and stop conditions into one minimal harness.

Why This Matters In Production

The loop is where an AI engineer proves the model is not directly controlling the system. The model proposes text. The harness decides whether to stop, reject, execute an allowed tool, append an observation, or end because the step budget is exhausted.

Invariant: the model requests; the harness validates, executes or rejects, records an observation, and decides whether another model call is allowed.

Predict Before Reading

The model returns {"tool": "submit", "args": {"answer": "done"}}, and the tools dictionary contains a callable named submit. Should the loop call that tool?

Prediction: no. submit is internal loop control. It is shaped like a request so the model can use one protocol, but the harness handles it as a stop condition.

Concrete Walkthrough

messages = [{"role": "user", "content": user_task}]

for step in range(max_steps):
    assistant_output = model.complete(messages)
    messages.append({"role": "assistant", "content": assistant_output})

    request = validate_tool_request(assistant_output)

    if request is invalid:
        messages.append(rejection_observation)
        continue

    if request.tool == "submit":
        return submitted_answer

    result = tools[request.tool](**request.args)
    messages.append(tool_observation(result))

return max_steps_exit

Failure Case

If the loop treats submit as a normal external tool, a final answer could trigger arbitrary code or be rejected as an unknown tool. If the loop fails to enforce max_steps, malformed output can spin forever. If it executes unknown tools, model text becomes authority.

Evidence And Observability

The first proof is test_submit_stops_loop. It asserts the loop calls the model once, does not execute read_file, does not execute a registered submit function, returns exit_reason == "submitted", and returns final_answer == "done".

Additional proofs cover tool observations, malformed JSON, unknown tools, invalid args, tool exceptions, and max_steps. Run them with uv run pytest curriculum/phase-1/06_agent_loop/tests_agent_loop.py.

Engineering Tradeoff

The loop can be strict and stop on the first invalid output, or it can return a rejection observation and let the model repair its next request. Project 1A chooses bounded recovery: invalid output becomes an observation, but max_steps prevents unlimited retries.

Practice: Choose The Harness Decision

The assistant returns malformed JSON on step 0, then a valid submit on step 1. What should the loop do on step 0?

The loop should append a structured INVALID_JSON observation, then continue if the step budget allows. That makes the failure visible to the next model call.
