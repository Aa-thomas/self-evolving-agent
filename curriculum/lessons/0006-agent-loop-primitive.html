Project 1A · Primitive 6 · Agent Loop

The agent loop is the harness's control flow

The model proposes text. The harness owns every transition that validates, classifies, executes, observes, recovers, and stops.

The model never operates the system directly

An assistant completion can contain a convincing request, but it is still untrusted text. The model cannot call read_file, append an observation, or end the run. Those are deterministic application decisions.

Central thesis: the model proposes the next move; the harness decides whether that proposal becomes a rejection, internal control flow, or an external tool invocation.

This division is the safety and reliability boundary. If execution were inferred from the model's wording, protocol validation, permissions, handler availability, and stopping policy would become suggestions rather than enforceable contracts.

The order matters. A rejected request cannot have a tool outcome because execution was never authorized. A submitted answer has no external tool result because submit is protocol-shaped internal control flow.

Walk one complete trajectory

Use the real scenarios in agent-loop-scenarios.json and the focused proof in tests_agent_loop.py.

  1. The initial message asks the agent to read notes.txt.
  2. The model emits a JSON read_file request. Parsing, registry lookup, and argument validation succeed.
  3. The loop classifies execution, calls the registered handler, and appends its structured result as a tool observation.
  4. The next model call receives that observation and emits a valid submit request.
  5. The loop classifies submission, returns the answer, and stops without invoking a runtime handler named submit.

Failures follow the same discipline. Malformed JSON and invalid arguments become observations that permit bounded recovery. A handler exception becomes TOOL_EXCEPTION context rather than escaping the loop. If no turn submits, max_steps ends the run even when every individual request was valid.

Commit the dispatch prediction

The protocol registry contains read_file; the runtime tools dictionary is empty. What should happen after validation succeeds?

The registry proves protocol support; it does not prove the runtime has a handler. Dispatch owns that second check.

One name, two registries, two failures

Compare test_unknown_protocol_tool_is_rejected_without_execution with test_unregistered_runtime_tool_is_rejected_without_execution. An unknown protocol tool fails before an action exists. A missing runtime handler fails after the request was accepted and execution selected.

unknown protocol tool
→ request rejected at tool lookup

known protocol tool, absent handler
→ request accepted → execute selected → runtime failure

The error codes preserve the boundary distinction: an unknown protocol name produces UNKNOWN_TOOL, while an accepted tool without an installed handler produces RUNTIME_TOOL_UNAVAILABLE. Their ordered variants also show that the failures occur at different transitions.

Run proof and bound the claim

Run uv run pytest curriculum/phase-1/06_agent_loop/tests_agent_loop.py. The focused tests establish covered transitions: submission stops, malformed output becomes an observation, tool results reach later calls, runtime drift is normalized, handler exceptions do not escape, and the budget stops repeated work.

They do not prove every tool is correct, every branch is instrumented, the model chooses useful actions, or arbitrary user tasks succeed. A green loop suite establishes control-flow mechanics, not product quality.
