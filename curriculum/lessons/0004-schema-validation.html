Validation is a four-gate checkpoint

The harness must reject bad model output before any tool code can run.

The Four Gates

Gate Rejects
JSON parse INVALID_JSON
Outer shape INVALID_TOOL_REQUEST_SHAPE
Tool registry UNKNOWN_TOOL
Tool args schema INVALID_TOOL_ARGS

Your validate_tool_request function should compose these gates in order. Each failure should return a structured rejection so the loop can later turn it into an observation.

Why This Is Not Tool Execution

Validation only answers: "May this request reach a tool handler?" It does not read files or write files. That separation is what lets unknown tools and bad arguments fail safely.

Practice: Classify The Failure

What error code should this produce: {"tool": "read_file", "args": {}}?

INVALID_TOOL_ARGS. The JSON is valid, the outer shape exists, and read_file is known, but the required path argument is missing.

Checkpoint

  • I can name the four validation gates in order.
  • I can distinguish wrong shape from wrong args.
  • I know unknown tools are rejected before execution.
