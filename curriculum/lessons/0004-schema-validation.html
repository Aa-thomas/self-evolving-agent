Primitive 4 · Schema Validation

Validation is a four-gate checkpoint

A request can be valid JSON and have the right outer shape while still naming an unknown capability or carrying arguments that violate that capability’s contract.

Structured is not the same as executable

Primitive 3 turns assistant text into a narrow ToolRequest. Its type proves that tool is a string and args is a JSON object. That is a transport guarantee: later code no longer has to guess whether it received prose, an array, or a half-finished JSON document.

It is not yet a capability guarantee. The parser does not know whether read_file belongs to this protocol, and it deliberately cannot know that read_file.path must be a string. Consequently, this request is correctly parsed but not executable:

ToolRequest(
    tool="read_file",
    args={"path": 123},
)
First principle: parsing establishes shared structure; validation establishes the contract of the specific operation before runtime execution.

If read_file is allowed to discover the bad type while opening a path, the failure has crossed too many boundaries. The runtime tool now has to duplicate protocol rules, failures arrive later, and different handlers may interpret the same malformed request differently.

Keep the four gates ordered

The phrase “valid tool call” hides four distinct claims. The request must cross them in order because each gate consumes a guarantee made by the one before it.

GateQuestionFailure evidenceOwner
1. JSONIs the assistant output one complete JSON document?INVALID_JSONparse_json
2. Request shapeIs the decoded value one strict ToolRequest envelope?INVALID_TOOL_REQUEST_SHAPEvalidate_tool_request_shape
3. Tool lookupDoes the protocol registry declare this tool name?UNKNOWN_TOOLvalidate_tool_args
4. Tool argumentsDo the arguments satisfy that tool’s strict model?INVALID_TOOL_ARGSThe selected argument schema

A later failure tells you which earlier claims succeeded. If a request reaches INVALID_TOOL_ARGS, it was valid JSON, had the required envelope, and named a registered protocol tool. This ordering turns an error code into causal evidence instead of a generic “bad request” label.

The invariant is precise: only a request for a registered tool whose arguments match its declared schema may leave this boundary as a ValidatedToolRequest.

Worked example: follow path=123

Consider the raw assistant output {"tool":"read_file","args":{"path":123}}. Walk it through the actual pipeline before deciding where it fails.

  1. JSON gate: the number 123 is legal JSON, so decoding succeeds.
  2. Request-shape gate: tool is a string and args is an object, so Primitive 3 returns ToolRequest.
  3. Tool-lookup gate: TOOL_SPECS["read_file"] exists and selects ReadFileArgs.
  4. Argument gate: strict validation requires path: str. It does not coerce the integer, so the result is InvalidToolArgsError(code="INVALID_TOOL_ARGS").

Now change only the tool name to delete_file. The request still crosses the first two gates, but registry lookup fails before any argument model runs. The correct evidence becomes UnknownToolError(code="UNKNOWN_TOOL").

Known name, bad arguments

read_file selects ReadFileArgs; schema validation rejects path=123 as INVALID_TOOL_ARGS.

Unknown name

delete_file selects no schema; registry lookup rejects it as UNKNOWN_TOOL.

The distinguishing evidence is not that both requests are “unsafe.” It is the first gate each request cannot cross.

One registry, one argument contract per capability

TOOL_SPECS maps each protocol name to a ToolSpec, and each spec names exactly one argument schema. This is more than a convenient lookup table. It makes two decisions inspectable at the same boundary: whether the protocol recognizes a name and what that name’s arguments mean.

ToolArgument contractImportant edge
read_filepath: strThe path is required and strict.
write_filepath: str, content: strBoth fields are required; extras are forbidden.
list_filespath: str | None = NoneAn empty object is intentionally valid.
submitanswer: strProtocol-valid control, not an external file operation.

The design tension is where to reject malformed arguments. A runtime-first design lets every handler validate for itself. A protocol-first design rejects before dispatch using a shared declared contract. This primitive chooses protocol-first validation because it produces consistent typed input and stable failure evidence while leaving domain checks—such as whether a path is permitted—to the runtime tool that owns them.

Strict models also forbid unknown fields. Accepting {"path":"notes.txt","random":true} would silently enlarge the executable protocol: the sender could believe random mattered while the handler ignored it. Rejection keeps the declared contract and observed behavior aligned.

Protocol validity still does not promise a handler

A protocol registry and a runtime handler dictionary answer different questions. The registry asks, “Is this operation part of the language, and are its arguments meaningful?” Runtime dispatch later asks, “Is executable code available for this operation in this process?”

That distinction is visible in submit. Its arguments can validate successfully, yet the agent loop should classify it as a control transition and stop instead of invoking a file-tool handler. Conversely, configuration drift could leave a protocol-valid external tool without a runtime handler. Reclassifying either case as a parser or schema failure would erase the actual bad transition.

Ownership rule: diagnose the first contract that fails. Do not move a later dispatch question backward merely because both outcomes prevent execution.

What the focused proof can support

The validator suite supports a bounded claim: for its covered cases, the pipeline distinguishes malformed JSON, invalid outer shape, unknown tool names, invalid arguments, and accepted typed requests. It specifically proves that path=123 is rejected at the tool-argument gate and that submit can be protocol-valid.

It does not prove that a path remains inside a sandbox, that a runtime handler is installed, that a handler succeeds, or that the loop interprets submit correctly. Those are different boundaries with different proof artifacts. A green validator test is necessary evidence for safe dispatch, not evidence of successful execution.

Commit a design decision

For ToolRequest(tool="read_file", args={"path":123}), which boundary owns the rejection?

Evidence: tool argument validation owns it and returns INVALID_TOOL_ARGS. The parser has already done its narrow job; deferring this to the runtime tool duplicates the protocol contract at the wrong boundary.

Build the narrow validation boundary

Implement or reconstruct validate_tool_args() while preserving the registry and result contracts:

  1. Look up request.tool in TOOL_SPECS before selecting an argument model.
  2. Return UnknownToolError when lookup fails; do not run a speculative schema.
  3. Validate request.args with the selected strict schema and normalize ValidationError into InvalidToolArgsError.
  4. Return a typed ValidatedToolRequest only after both checks succeed.

Leave JSON parsing in Primitive 3. Do not open files, dispatch handlers, merge runtime availability into TOOL_SPECS, or special-case submit control flow here.
