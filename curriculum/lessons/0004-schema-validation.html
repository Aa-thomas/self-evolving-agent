Project 1A Primitive 4

Validation decides what may reach a tool

The harness must reject bad model output before any permissioned code path can run.

Why This Matters In Production

When a model emits a tool request, the service needs a deterministic gate between untrusted text and executable code. This is where an AI reliability engineer prevents accidental tool calls, confusing errors, and incident reports with no clear cause.

Invariant: validation returns a structured accepted request or a structured rejection. It never executes the requested tool.

Predict Before Reading

Classify this request: {"tool": "read_file", "args": {}}. Is it malformed JSON, wrong outer shape, unknown tool, or invalid args?

Prediction: INVALID_TOOL_ARGS. The JSON decodes, the outer object has tool and args, and read_file is known, but the required path argument is missing.

Concrete Walkthrough

Gate Rejects Owns
JSON parse INVALID_JSON Can the text decode?
Outer shape INVALID_TOOL_REQUEST_SHAPE Is it a request object?
Tool registry UNKNOWN_TOOL Is this tool allowed here?
Tool args schema INVALID_TOOL_ARGS Can this tool receive these args?

The focused validation proof lives in validate_tool_args_tests.py. The loop-level proof lives in tests_agent_loop.py, where invalid requests become observations instead of executions.

Failure Case

The assistant asks for {"tool": "delete_file", "args": {"path": "notes.txt"}}. If the harness only checks that JSON has tool and args, it may route an unsupported action toward code that should not exist. The registry gate must stop it with UNKNOWN_TOOL.

Evidence And Observability

Evidence must show both rejection and non-execution. A good test asserts the error code and then checks the fake tool's call list is empty. That proves the harness did not merely notice the problem after side effects happened.

Run the focused loop proofs with uv run pytest curriculum/phase-1/06_agent_loop/tests_agent_loop.py -k "unknown or invalid".

Engineering Tradeoff

Strict validation can reject requests that a human could repair. That is acceptable here because the loop can return a structured observation and let the model try again. The tradeoff is strictness versus recovery; Project 1A chooses strict gates plus bounded retry.

Practice: Pick The Gate

The assistant returns {"tool": "read_file"}. Which gate should reject it?

The request has valid JSON but the wrong outer shape. It should return INVALID_TOOL_REQUEST_SHAPE.
