Project 1A Primitive 8

Evals define what working means

An eval runner turns representative tasks into repeatable pass/fail evidence, so the agent loop is judged by behavior instead of hope.

Why This Matters In Production

A single unit test can prove one branch. It cannot prove the harness works as a small system. An eval suite gives an applied AI engineer a defensible answer to: "Did the change make the agent better, worse, or just different?"

Invariant: every eval case should specify the task, setup, scripted model behavior or fixture, expected outcome, and evidence path for failure analysis.

Predict Before Reading

A case asks the agent to copy one file to another. Is that only a write_file unit test?

Prediction: no. It is a composed behavior: read a source file, append the observation, write a target file, append that observation, then submit the answer.

Concrete Walkthrough

{
  "name": "copy-file",
  "user_task": "Copy a.txt to b.txt.",
  "sandbox": {"a.txt": "hello"},
  "model_outputs": [
    "{\"tool\":\"read_file\",\"args\":{\"path\":\"a.txt\"}}",
    "{\"tool\":\"write_file\",\"args\":{\"path\":\"b.txt\",\"content\":\"hello\"}}",
    "{\"tool\":\"submit\",\"args\":{\"answer\":\"copied\"}}"
  ],
  "expected_file": {"b.txt": "hello"},
  "expected_exit_reason": "submitted"
}

This branch does not yet provide a dedicated eval runner implementation file. Treat this lesson as the design contract for the runner you will later add. The existing loop proofs in tests_agent_loop.py show the behaviors the future eval suite should compose.

Failure Case

A refactor still passes test_submit_stops_loop but breaks the copy-file flow because observations are appended with the wrong role. A unit proof catches the stop condition; an eval catches the user-visible workflow. You need both.

Evidence And Observability

A future eval result should include total cases, passed cases, failed cases, per-case reason, and a trace or trajectory reference. Since that runner is not implemented in this branch, do not claim eval artifacts exist yet. The immediate evidence remains the focused unit tests for the loop.

Run the current behavioral proof set with uv run pytest curriculum/phase-1/06_agent_loop/tests_agent_loop.py.

Engineering Tradeoff

Small eval suites are fast and easy to understand, but they miss edge cases. Large suites catch more regressions, but they can slow iteration and make failures noisy. Project 1A should start with a small deterministic suite that covers each harness invariant once.

Practice: Unit Test Or Eval?

Which case belongs in a Project 1A eval suite rather than only as a single-function unit test?

An eval case should prove the loop works in context. A copy-file task requires multiple model outputs, multiple tool observations, final grading, and failure evidence.

Transfer Prompt

If a customer cared about "never modify files without confirmation," what eval case would you add before changing the loop?

Checkpoint

  • I can explain what fields an eval case needs.
  • I can distinguish a unit proof from an eval proof.
  • I know not to promise eval artifacts before the repo writes them.
