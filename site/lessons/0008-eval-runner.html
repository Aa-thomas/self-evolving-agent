Project 1A · Primitive 8 · Eval Runner

Evals define what working means

Build the smallest eval runner that can distinguish a successful composed workflow from a nearly identical failure—and preserve enough evidence to explain the difference.

A green unit suite is not the decision

The parser, validator, loop, and trace logger each have focused tests. Those tests are necessary, but the user does not ask for an isolated transition. The user asks the harness to copy a.txt to b.txt and report completion across three model outputs, two tool calls, message-state updates, and a final submit.

Decision question: does this composed harness satisfy the declared copy-file behavior for the configured trajectories, or do its component tests merely make that outcome plausible?

First principle: an eval supports a scoped behavioral claim by comparing defined cases against explicit outcomes under controlled conditions. One successful run is an example; a repeatable case plus a grader is evidence.

State the claim before looking at the score

Hypothesis: if each configured case runs with fixed model outputs, tools, step budget, and expected fields, then the runner can determine whether the actual trajectory satisfies every declared outcome and retain causal evidence for a mismatch.

Scope: the deterministic Project 1A cases in this repository. This does not claim a production success rate, generalization to unlisted tasks, statistical significance, or reliable behavior from an arbitrary provider.

Unit of evaluation

One complete named trajectory: task, fixed model outputs, tools, budget, expected result, actual result, and trace path.

Pass condition

Every declared expected field equals the corresponding actual field. Tool success alone is not the contract.

Failure evidence

Expected and actual records, the first named mismatch, and the trace path remain attached to the failed case.

Control the comparison or lose attribution

Keep fixedWhy
Case input and expected behaviorA changed contract cannot explain whether the implementation improved.
Fake-model outputs and toolsChanging the trajectory and the evaluator together creates competing causes.
max_steps and grading ruleA different budget or pass condition changes what the case measures.

Confounder: if you change the model outputs, tool implementation, expected answer, and evaluator in the same comparison, a different result cannot be attributed to one boundary. Likewise, an aggregate score without per-case evidence can hide which behavior changed.

Commit the wrong-answer outcome

Two runs use the same task, tools, step budget, file side effect, and expected record. Both write b.txt="hello" and exit through submit. The second run submits "wrong" instead of the expected "copied". How should the configured eval classify the second run?

Measured outcome: fail. The declared final_answer is "copied"; the actual value is "wrong". The successful tools establish other fields, but they do not erase this mismatch.

The aggregate result is 1 / 2, but that count is not the diagnosis. The failed EvalCaseResult must preserve its expected record, actual record, focused failure reason, and traces/wrong-answer.json so the next investigation starts from evidence.

Choose the measurement design deliberately

One boolean and a total

Compact, but it loses the mismatched field and causal path. A regression becomes a score with no responsible transition.

Per-case evidence, then totals

More structure, but the aggregate is derived from inspectable case results. This primitive chooses that design.

Decision rule: preserve the smallest record that lets an engineer reproduce the case, identify the violated expected field, and follow the trace. Do not solve trace collection again inside the evaluator; retain its path.

Build the runner, not the answer key

Open the real starter eval_runner.py. Its dataclasses define the reporting boundary; the two function bodies are deliberately withheld.

  1. Implement evaluate_case: run one case once, compare every declared expected field, and return an EvalCaseResult.
  2. On failure, name the mismatched field in failure_reason and preserve both records plus trace_path.
  3. Implement run_eval_suite: evaluate every case and derive total, passed, and failed from the case results.

Leave the agent loop, trace logger, fixture expectations, fake-model outputs, tools, and step budget unchanged. Do not special-case copy-file, mutate expected values to match actual output, call one successful run a suite, or collapse failures into only a score.

The starter proof is intentionally red. Run uv run pytest curriculum/phase-1/08_eval_runner/tests_eval_runner.py. The fixture test already passes; test_eval_runner_reports_each_case_and_causal_trace fails at NotImplementedError until you build the runner. That red run proves the handoff is real, not that the lesson is complete.

Practice measurement decisions

For each case, choose the evidence or interpretation that keeps the behavioral claim attributable and inspectable.

Which evidence should first prove that validate_tool_args rejects an integer path?

Which evidence should grade “copy a.txt to b.txt and report completion”?

A new run changes model outputs, tools, max_steps, and expected fields, then scores higher. What conclusion is justified?

The file content and exit reason match, but final_answer does not. How does this case affect the configured claim?

A report says failed=1 but stores no case result, mismatch, or trace path. What evidence quality does it provide?

A real trajectory loses a tool observation and produces the wrong file. After diagnosis, what should preserve the failure?

Run proof, inspect failure evidence, and bound the claim

Run uv run pytest curriculum/phase-1/08_eval_runner/tests_eval_runner.py. When it turns green, inspect both EvalCaseResult records rather than stopping at passed=1 and failed=1.

The proof establishes

The runner grades every declared field for the two covered deterministic trajectories, reports each outcome, preserves a focused wrong-answer failure reason and trace path, and derives accurate totals.

The proof does not establish

Production success rates, statistical significance, coverage of unlisted tasks, reliable provider behavior, or that every future failure will have enough trace evidence.

Interpret before claiming improvement: state whether the evidence supports, weakens, or leaves the configured claim unresolved. Cite one expected-versus-actual field and its trace path. Then name the broadest claim you are still unwilling to make.

Transfer: redesign the measurement

Add a new behavior: “Never modify a file until the user has confirmed the write.” Decide which task and expected fields belong in the new eval case, which fake outputs, tools, and budget must stay fixed, whether the permission boundary also needs a focused unit regression, and what a failed result must preserve for diagnosis.
