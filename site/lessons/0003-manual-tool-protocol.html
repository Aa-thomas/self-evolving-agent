Project 1A · Primitive 3

Turn one assistant string into one ToolRequest

The parser owns JSON decoding and the outer request shape. It does not decide whether a tool exists or whether its arguments are safe.

Start from the prior primitive

Primitive 2 stores the assistant output as a message. Inspect message_state.py: its job is to preserve that output, not interpret it. The next boundary receives the raw text and must either produce the ToolRequest defined in 03_parse_tool_request.py or a structured rejection.

Invariant: exactly one JSON object with tool and args becomes a request. No informal syntax, extra prose, registry lookup, or argument schema is accepted here.

Commit a boundary prediction

{"tool":"read_file"} is valid JSON but lacks args. Which result should the parser return?

It is INVALID_TOOL_REQUEST_SHAPE: decoding succeeded, but ToolRequest.model_validate cannot establish the required outer shape. Tool names belong to Primitive 4.

Build and prove the narrow parser

Implement or reconstruct parse_tool_request(). Run uv run pytest curriculum/phase-1/03_parse_tool_request_tests.py. The proof covers test_valid_request_produces_tool_request and test_trailing_prose_is_not_json; it establishes the parser boundary, not executable tool availability.

Explain the proof, then transfer

After the tests pass, inspect your implementation: where can INVALID_JSON originate, and where can it not? The suite proves text-to-request behavior; it does not prove read_file is registered or that path is a string. For a changed protocol that permits a top-level request_id, decide whether that field belongs in this outer-shape boundary or the next validation boundary.
