Project 1A Primitive 3

A tool call starts as untrusted JSON text

Before schemas, tools, or loops, the harness must parse one assistant string without pretending the model executed anything.

Why This Matters In Production

Agent systems often fail because teams treat model output as an action instead of a request. The parser is the first trust boundary. It converts text into data or returns a structured rejection that the rest of the harness can handle.

Invariant: valid JSON is not the same as a valid tool request. Primitive 3 proves only that the assistant text can be decoded as JSON.

Predict Before Reading

Which of these should produce INVALID_JSON: {"tool": "read_file"} or {"tool": "read_file", "args": ?

Prediction: only the second. The first is valid JSON with the wrong Project 1A shape; that is a later validation failure.

Concrete Walkthrough

raw = '{"tool": "read_file", "args": {"path": "notes.txt"}}'
json_result = parse_json(raw)

if not json_result.ok:
    return INVALID_JSON

data = json_result.value

The parser entry point is 03_parse_tool_request.py. At this stage, the important behavior is narrow: malformed text becomes INVALID_JSON; decoded JSON moves forward as data.

Failure Case

A model returns read_file("notes.txt"). A human understands the intention, but json.loads must reject it. Accepting informal formats creates a support burden: every future prompt wording becomes an unofficial protocol.

Evidence And Observability

The evidence is a small parser proof: malformed JSON returns a structured result with ok=False and error_code="INVALID_JSON". The later agent-loop proof test_malformed_json_is_rejected_without_execution confirms the bad text does not reach a tool.

Run the composed proof with uv run pytest curriculum/phase-1/06_agent_loop/tests_agent_loop.py -k malformed_json.

Engineering Tradeoff

Strict JSON feels less forgiving than natural-language recovery, but it makes the protocol testable. For Project 1A, choose strict parsing and let the loop return an observation so the model can try again within max_steps.

Practice: Classify The Bad Output

The assistant returns "hello". What should primitive 3 say?

"hello" is valid JSON. It is not a valid Project 1A tool request, but that is primitive 4's job.

Transfer Prompt

If a customer wanted XML or YAML tool requests, what would have to change in the parser while keeping the later validation boundary intact?

Checkpoint

  • I can write the one-object JSON tool-request shape from memory.
  • I can explain why the model is making a request, not executing code.
  • I can distinguish malformed JSON from valid JSON with the wrong shape.
