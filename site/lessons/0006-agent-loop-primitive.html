Project 1A Primitive 6

The agent loop is the harness's control flow

Primitive 6 composes the first five primitives: call the model, treat messages as state, parse untrusted tool JSON, validate it, execute only allowed tools, and stop safely.

Why This Matters

The curriculum's thesis is: hold the model fixed and treat the harness as the learnable surface. In Project 1A, the loop is where that becomes real. The model returns text. Your harness turns that text into a decision.

Working mental model: the model requests; the harness validates, executes or rejects, records an observation, and decides whether to continue.

OpenAI's function-calling docs describe tool calling as a multi-step conversation: your application sends a request, receives a tool call, executes application-side code, returns the tool output, then gets a final response or more tool calls. Project 1A rebuilds that flow manually so you can see each moving part.

The Loop Shape

1. Ask model

Call model.complete(messages). The model sees the current message state.

2. Parse action

Treat assistant output as untrusted JSON and convert it into a tool request.

3. Decide

If the request is submit, stop. If invalid, reject. If known and allowed, execute.

4. Append observation

Add the result to messages, then let the next model call react to it.

Keep these references open while coding: Agent Loop Contract and Model Completion Boundary.

The Algorithm

messages = [user task]

for step in range(max_steps):
    assistant_output = model.complete(messages)
    append assistant message
    request = validate_tool_request(assistant_output)

    if invalid:
        append rejection as tool observation
        continue

    if request.tool == "submit":
        return submitted result

    tool_result = tools[request.tool](**request.args)
    append tool_result as tool observation

return max_steps result

The implementation target is 06_agent_loop/agent_loop.py. Keep this file focused on the control loop: model call, validation, submit, tool execution, observation, and max-step exit.

Start From The First Test

Your current active proof is test_submit_stops_loop. It says:

model returns: {"tool": "submit", "args": {"answer": "done"}}

run_agent should:
- call the model once
- not execute read_file
- return exit_reason == "submitted"
- return final_answer == "done"

This is the smallest useful loop because it proves submit is a stop condition, not a normal tool execution.

Stop Conditions

submit

Returns exit_reason="submitted" and the submitted answer. No external tool is executed.

max_steps

Returns exit_reason="max_steps" when the model never submits in the allowed budget.

Practice: Retrieve The Contract

Without looking above, write the four moving parts the loop needs after the user task enters the system.

Model call, parse and validate assistant output, stop on submit or execute/reject a tool, append an observation before the next model call.

Why must read_file not be called in test_submit_stops_loop?

submit is the loop's stop signal. It is shaped like a tool request so the model can use the same protocol, but the harness handles it internally and returns the final answer without executing an external tool.

Why does invalid JSON become a tool observation instead of ending the whole run immediately?

The loop records the rejection as an observation so the next model call can see what failed and try a valid action, while max_steps still prevents infinite retries.
