Project 1A Primitive 1

A model call is a boundary, not a brain

The first primitive turns one user prompt into one raw assistant string and records enough evidence to debug that call later.

Why This Matters In Production

An applied AI engineer is responsible for the seam between provider behavior and application behavior. If that seam is blurry, every later bug becomes harder to assign: did the model say the wrong thing, did the harness parse it wrong, or did a tool run when it should not have?

Invariant: call_model asks a model for text and returns a structured record. It does not parse tool JSON, execute tools, or decide whether an answer is final.

Predict Before Reading

A fake model returns {"tool": "read_file", "args": {"path": "notes.txt"}}. Should primitive 1 reject that as a tool request?

Prediction: no. At this layer it is only output text. Rejection belongs to the parser and validation layers that come later.

Concrete Walkthrough

record = call_model(prompt="ping", model=model)

record.output_text
record.latency_seconds
record.input_tokens
record.output_tokens
record.usage_source

The model object only needs .complete(messages). The primitive builds one message list, calls the model, measures latency, confirms the return value is a string, and stores provider token usage or a clear estimate.

Implementation and proof live in 01_model_call.py and tests_01_model_call.py. The key proof is that a fake model receives exactly one user message and returns a ModelCallRecord.

Failure Case

Suppose model.complete returns {"content": "pong"} instead of "pong". If primitive 1 accepts that object, later code may treat provider-specific shapes as normal harness state. The boundary has leaked.

The correct behavior is a clear type failure at the model boundary. That makes the incident reproducible before the output reaches parsing, validation, or tools.

Evidence And Observability

Convincing evidence for this primitive is not a demo response from a live API. It is a deterministic test with a fake model:

  • the fake model records the exact message list it received,
  • record.output_text equals the fake output,
  • record.latency_seconds exists,
  • token usage is either provider-sourced or explicitly estimated.

Run the focused proof with uv run pytest curriculum/phase-1/01_model_call/tests_01_model_call.py.

Engineering Tradeoff

Keep the adapter thin even though it feels convenient to parse JSON here. A thin adapter is less magical and easier to test. The tradeoff is that later primitives must do more explicit work, but that separation makes failures easier to localize.

Practice: Choose The Boundary

A provider returns raw text that happens to look like a tool request. What should call_model do?

call_model should return a record containing the raw assistant text, latency, and token usage or estimate. Later primitives decide whether that text is JSON, a valid request, or a final answer.

Transfer Prompt

If a customer required latency SLO reporting, what extra field would you add to ModelCallRecord without letting provider details leak into the agent loop?

Checkpoint

  • I can explain why complete returns raw text.
  • I can point to the evidence fields in ModelCallRecord.
  • I know that parsing and tool execution do not belong in primitive 1.
