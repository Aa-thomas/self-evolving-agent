Primitive 1 · Model-Call Boundary

A model call is text in, text out, plus evidence

A model gives your program data. Before an agent can reason about that data, one small boundary must make its shape, input, and evidence explicit.

Why begin here?

An LLM call is not an action. It is a request that returns provider-shaped data. If every later part of the harness reaches into that provider response directly, a provider change becomes a system-wide bug and a failed run becomes hard to explain.

First principle: the model returns data; the harness owns the boundary that checks, records, and normalizes that data before anything else uses it.

This primitive does not parse tool requests, run tools, retry failures, or maintain an agent conversation. It establishes the smallest useful contract: one call in, assistant text plus evidence out.

Start with the evidence, not the finished solution

Before you write call_model, inspect the real tests_01_model_call.py. Read FakeModel, FakeModelWithUsage, and BadModel. These are not invented lesson examples: they are the proof artifacts for this primitive.

Walk through one ordinary call

Suppose the caller supplies prompt="ping". The boundary first creates one message: {"role": "user", "content": "ping"}. It sends that exact list to the model once. When the fake provider returns "pong", the boundary records the text, the elapsed time, and either provider usage or clearly labeled estimates.

StepFact that becomes trueWhy it matters later
1. Normalize inputThere is one ordered message list.A later test can prove exactly what the model saw.
2. Complete onceThe provider returns one value and optional usage.Timing and usage have a single collection point.
3. Enforce textThe internal completion is a string.Later parser code never needs provider-object branches.
4. Return a recordOutput and evidence travel together.A run can be inspected without re-running the provider.

Now contrast the failure

BadModel returns {"content": "not a string"}. It may look tempting to pull out content and keep going. That would silently invent a second provider contract at the wrong layer.

Coerce it

The loop continues today, but later code now depends on a provider-specific object shape that the record never promised.

Reject it at call_model

The failure is early, honest, and local: the provider boundary violated the internal text contract.

The second choice wins here. A narrow boundary should not hide a changed provider response just to make a downstream feature appear to work.

The boundary makes a bounded claim

A deterministic fake model can prove what messages crossed this boundary, what value came back, and how the harness normalized that value. It cannot prove that a live provider will be available, that its text is useful, or that a later agent loop will choose a sound action. This separation is not cautionary boilerplate: it prevents a local contract test from being misreported as end-to-end reliability evidence.

The durable rule is to make the strongest claim supported by the observed artifact—and stop there. That is why the record carries both output and provenance while the provider adapter owns any provider-specific response shape.

Build from the scaffold

Open the real model_call_skeleton.py, not the completed target. Copy it into your working model_call.py and implement only call_model.

  1. Require exactly one of prompt or messages.
  2. Turn a prompt into one user message while preserving a supplied message list.
  3. Call the model once, time it, and reject non-string output.
  4. Return one record with optional provider usage or explicit estimates.

Leave tool parsing, retries, and conversation state untouched. This is a provider boundary, not an agent loop.

Run proof, then make a bounded claim

Run uv run pytest curriculum/phase-1/01_model_call/tests_01_model_call.py. Read the output alongside test_call_model_sends_single_user_message_and_returns_output_record and test_call_model_rejects_non_string_model_output.

The proof establishes

For the covered fake providers, the boundary sends the expected input, returns text in a record, records usage provenance, and rejects a non-string completion.

The proof does not establish

That a real provider is reliable, a tool is safe, or a multi-step agent will recover from a bad trajectory.

Explain: In two or three sentences, connect the non-string test to the boundary that owns the correction. Then say why the same test cannot prove a real provider will never change shape.

Transfer the reasoning

A second provider returns a rich response object with text, token counts, and a provider request ID. Decide which facts belong in the record, which must be normalized into the internal text contract, and which must stay inside a provider adapter. Name one regression assertion that would keep that decision honest.

Ask your teacher to review your boundary claim and the one proof assertion you chose.
