Project 1A Primitive 2

Message state is the model's working memory

There is no hidden memory in Project 1A. The model knows only the messages your harness sends on the next call.

Why This Matters In Production

When an agent takes multiple steps, operators need to know why the second step happened. The answer should be visible in the message list. If a tool result is not appended, the next model call may guess, repeat work, or fabricate progress.

Invariant: every observation the model should use later must be represented as a message before the next model.complete(messages) call.

Predict Before Reading

A model first asks to read notes.txt. The tool returns "file contents". Before the second model call, what roles should be present?

Prediction: user, then assistant, then tool. The second call can only react to the file contents if the tool observation is in that list.

Concrete Walkthrough

messages = [
  {"role": "user", "content": "Read notes.txt."},
  {"role": "assistant", "content": "{\"tool\":\"read_file\", ...}"},
  {"role": "tool", "content": "{\"ok\": true, \"value\": \"file contents\"}"}
]

assistant_output = model.complete(messages)

The current simple implementation is message_state.py. The composed proof appears later in test_tool_result_is_appended_before_next_model_call, where the fake model records the exact messages passed into the second call.

Failure Case

If the harness executes read_file but forgets to append the result as a tool message, the next model call cannot know the file contents. It may call read_file again, submit an unsupported answer, or invent contents. This is a reliability failure, not a model-memory mystery.

Evidence And Observability

The useful proof inspects the second model call, not the final answer. In tests_agent_loop.py, the fake model captures model.calls[1]. That call should contain roles in this order: user, assistant, tool, and the tool message should decode to the structured observation.

Run the focused proof with uv run pytest curriculum/phase-1/06_agent_loop/tests_agent_loop.py -k tool_result.

Engineering Tradeoff

Passing the full message list is simple and auditable. The tradeoff is context growth. In Project 1A, choose explicit full state because the system is small and you are learning the causal chain. Summarization is a later optimization, not a primitive-2 shortcut.

Practice: Diagnose The State Bug

The second model call repeats read_file even though the tool succeeded. Which fix should you check first?

The model sees only the passed message list. If a tool result is missing from that list, the next call has no reliable way to use it.

Transfer Prompt

If the task involved private customer data, what would you include in message state for reasoning, and what would you keep out for privacy or audit reasons?

Checkpoint

  • I can list the message roles used in Project 1A.
  • I can explain why memory is passed context, not magic storage.
  • I can test that a tool observation reaches the next model call.
