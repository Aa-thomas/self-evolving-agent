Project 1A Primitive 7

A trace turns a run into evidence

A trace is the causal record an engineer uses to explain what the model said, what the harness understood, what the tool returned, and why the run stopped.

Why This Matters In Production

When an agent gives the wrong answer, a user will not accept "the model did something weird" as an incident report. A trace lets you separate model behavior from harness behavior and make a defensible fix.

Invariant: every step should preserve the assistant output, parsed action or rejection, tool result or rejection, and stop reason. Without that chain, debugging becomes guesswork.

Predict Before Reading

A run ends with UNKNOWN_TOOL. Which field should help you decide whether the model asked for the wrong tool or the harness registry was missing a valid tool?

Prediction: the trace needs both assistant_output and the structured rejection. One shows what the model requested; the other shows how the harness classified it.

Concrete Walkthrough

{
  "step": 0,
  "assistant_output": "{\"tool\":\"delete_file\",\"args\":{\"path\":\"notes.txt\"}}",
  "parsed_action": {"tool": "delete_file", "args": {"path": "notes.txt"}},
  "observation": {"ok": false, "error_code": "UNKNOWN_TOOL"},
  "exit_reason": null
}

This branch does not yet provide a dedicated trace logger implementation file. Treat this lesson as the contract for the trace you will later add: it must preserve the same causal sequence already asserted by tests_agent_loop.py.

Failure Case

A failed run only records "final_answer": null. That is not enough. You cannot tell whether the model emitted malformed JSON, the parser rejected the shape, the tool threw an exception, or max_steps ended a recoverable loop. The trace must capture the decision points, not only the final result.

Evidence And Observability

Until the trace logger exists, the closest available evidence is the fake model call history in tests_agent_loop.py. A future trace should make those hidden assertions visible as a saved artifact: roles sent to the model, assistant output, validation outcome, tool observation, and exit reason.

Do not claim Project 1A has trace artifacts until the repository writes them. For now, use this lesson to define what the artifact must contain.

Engineering Tradeoff

Detailed traces improve debugging and auditability, but they can store sensitive content. Project 1A should start with full local traces because the sandbox is local and educational. A production system would need redaction, retention limits, and access control.

Practice: Diagnose The Missing Field

A trace stores exit_reason="max_steps" but not the assistant outputs. What debugging question becomes impossible to answer?

A max_steps result without assistant outputs cannot show whether the model was stuck repeating valid tool calls, emitting invalid JSON, or drifting between unrelated actions.

Trace Lab: Diagnose The Decision

Use the visible fields to distinguish what the model proposed from what the harness decided.

Transfer Prompt

If traces contained customer files, which fields would you redact, hash, or store behind stricter access controls?

Checkpoint

  • I can name the fields needed to replay one agent step.
  • I can use a trace to separate model output from harness parsing.
  • I know not to promise trace artifacts before the repo writes them.
