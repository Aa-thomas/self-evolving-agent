Project 1A Primitive 7

A trace is replayable evidence

A run without a trace is hard to debug. A trace makes every model action, parsed action, tool result, and exit reason inspectable later.

The JSON Shape

{
  "exit_reason": "submitted",
  "final_answer": "ok",
  "messages": [...],
  "steps": [
    {
      "step": 0,
      "assistant_output": "...",
      "parsed_action": {"tool": "write_file", "args": {...}},
      "tool_result": {"ok": true, "value": {...}},
      "exit_reason": null
    }
  ]
}

Your implementation target is the seventh primitive, 07_trace_logger.py. The useful helpers to design are trace_to_dict and write_trace.

What To Look For First

  • assistant_output: what the model actually said.
  • parsed_action: what the harness understood.
  • tool_result: what execution or rejection produced.
  • exit_reason: why the run stopped.

Practice: Debug From A Trace

If assistant_output is malformed JSON and parsed_action is null, which field tells you the exact rejection?

tool_result contains the structured rejection, including fields such as ok, error_code, and error.

Checkpoint

  • I can name the five required fields in a trace step.
  • I can use a trace to separate model output from harness parsing.
  • I know every eval result should point to a trace path.

Primary source: Local Harness Engineering curriculum, especially Project 1A's evidence artifact requirement. Proof to create: a trace JSON file with every required step field.

Ask your agent teacher to read a failed trace with you before changing code.
